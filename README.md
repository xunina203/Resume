## 给后续 Agent 的工作规则

### 1. 开始前先确认范围

- 先看用户当前要面试的公司、岗位、JD 或聊天上下文。
- 先读基础简历和对应公司文件夹里的已有文档，再写新内容。
- 如果没有 JD，就基于用户已给出的岗位名称做合理假设，并在文档开头标明假设。
- 不要默认修改 PDF 简历本体。除非用户明确要求编辑简历文件，否则先写 Markdown 修改建议。

### 2. 新公司/新岗位的推荐文件结构

每个公司文件夹建议包含以下文件：

- `[公司][岗位]面试准备.md`：业务背景、岗位理解、关键术语、高频问题。
- `简历匹配与修改建议.md`：是否需要改简历、怎么改、哪些经历要前置。
- `模拟面试题库.md`：按 HR、业务主管、专业技术、英文面试等轮次组织问题。
- `面试复盘.md`：面试后记录真实问题、回答效果、下次改进点。

### 3. 简历分析规则

- 只能基于用户简历和用户确认过的信息写经历，不要编造公司、岗位、项目或数据。
- 可以重写表达方式，但不能改变事实和数字含义。
- 每次简历建议都要回答三个问题：是否需要修改、为什么要改、具体改哪几处。
- 面向供应链、计划、物控、项目管理岗位时，优先突出 OTD、MPS、CTB、BOM、NPI 到 MP、库存周转、呆滞物料、供应商交付、SAP/SRM 等关键词。
- 如果目标岗位和过往行业不同，要明确写出“可迁移能力”，同时准备行业差异的回答。

### 4. 面试准备规则

- 输出要直接服务面试，不写空泛鸡汤。
- 每个岗位至少准备：一分钟自我介绍、岗位匹配主线、三个 STAR 案例、十个高频问题、三个反问面试官的问题。
- 对专业问题，回答要有结构：先给结论，再给方法，再给案例或指标。
- 对模拟面试，建议一次只问一个问题，等用户回答后再评分、追问、改写示范答案。
- 默认使用中文。如果岗位可能涉及英文沟通，再补英文自我介绍和英文问答。

### 5. Git 和文件规则

- GitHub 远程默认使用 SSH，不改回 HTTPS。
- 不提交 `.DS_Store`、临时文件、截图缓存或导出的中间文件。
- 修改前先看 `git status`，不要覆盖用户未提交的改动。
- 用户要求“做完提交推送”时，完成文档后执行提交并推送到当前分支。
- 提交信息要简洁说明本次新增或修改内容。
